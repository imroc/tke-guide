# Why Native Routing (GR) node pools must add the cilium agent-not-ready taint

## Background

After installing cilium on a TKE cluster and creating new node pools, if you're using the **Native Routing (GR)** option, you **must** add the following taint to the node pool:

```text
node.cilium.io/agent-not-ready=true:NoSchedule
```

The other three modes (Native Routing VPC-CNI, Overlay VPC-CNI, Overlay GR) do **not** need this taint. This document explains the difference.

## Symptoms

Without this taint, newly joined nodes may exhibit:

- Some Pods on the node (especially early-scheduled ones like `csi-cbs-controller` or components that depend on the metadata service) repeatedly CrashLoopBackOff due to networking issues
- Pods miss cilium-provided capabilities like masquerade and NetworkPolicy
- Restarting the node partially recovers things, but the situation is unstable

The root cause is a **CNI configuration timing race** at node-join time.

## Why GR mode's CNI is special

In GR mode:

- **Every node has a different PodCIDR** (GR allocates a subnet per node as that node's PodCIDR)
- The CNI configuration is dynamically generated per-node by `tke-bridge-agent`, **containing that node's specific subnet info**

```text
Node A (PodCIDR 10.244.1.0/24)         Node B (PodCIDR 10.244.2.0/24)
┌─────────────────────────┐            ┌─────────────────────────┐
│ /etc/cni/net.d/         │            │ /etc/cni/net.d/         │
│  10-tke-bridge.conflist │            │  10-tke-bridge.conflist │
│  ┌───────────────────┐  │            │  ┌───────────────────┐  │
│  │ subnet:           │  │            │  │ subnet:           │  │
│  │   10.244.1.0/24   │  │            │  │   10.244.2.0/24   │  │
│  └───────────────────┘  │            │  └───────────────────┘  │
└─────────────────────────┘            └─────────────────────────┘
```

This means cilium **cannot use a unified CNI config to take over all nodes** the way VPC-CNI or Overlay modes do. It can only watch tke-bridge's generated CNI config via `chainingTarget` and append itself at the tail of the chain.

## The timing race

When a node joins the cluster, the desired ordering is:

1. tke-bridge-agent writes the CNI config (containing this node's PodCIDR info)
2. cilium agent starts up, detects the tke-bridge config, appends itself to the chain
3. kubelet sees the full CNI chain is ready and begins scheduling business Pods
4. Pods go through the tke-bridge + cilium-cni enhanced chain, getting masquerade, NetworkPolicy, etc.

But what actually happens is:

```text
T0: Node joins cluster
T1: tke-bridge-agent writes CNI config ────┐
T2: kubelet sees CNI ready, schedules Pods │ Timing race: cilium hasn't appended yet!
T3: Pods start with "bare tke-bridge CNI" ─┘
T4: cilium agent finishes starting, appends to chain
T5: Only newly-created Pods get cilium enhancement
```

Pods created in the T2 → T3 window end up in a "broken" state:

- Their network config comes from bare tke-bridge, without cilium-cni enhancement
- Missing masquerade — they may fail to reach the TKE metadata service, etc.
- Missing NetworkPolicy enforcement
- Even after cilium agent starts, these Pods have already "missed" cilium-cni initialization and won't auto-recover

## What the taint does

When you add `node.cilium.io/agent-not-ready=true:NoSchedule` to the node pool:

- Newly joined nodes carry this taint **by default**, so no business Pods get scheduled there yet
- After cilium agent finishes starting, **it automatically removes this taint** (built-in cilium behavior)
- Only then does kubelet start scheduling business Pods — and the CNI chain is already ready, so Pods are in a "complete" state

```text
T0: Node joins (with agent-not-ready taint, no business Pods scheduled)
T1: tke-bridge-agent writes CNI config
T2: cilium agent starts → appends to chain → auto-removes taint
T3: Business Pods start being scheduled, CNI chain complete ✓
```

## Why other modes don't need it

| Mode                     | Needs taint     | Reason                                                                                         |
| ------------------------ | --------------- | ---------------------------------------------------------------------------------------------- |
| Native Routing (VPC-CNI) | ❌ Not needed   | `cni.customConf=true`, one shared ConfigMap, no per-node dynamic generation → no race          |
| Native Routing (GR)      | ✅ **Required** | Different PodCIDR per node, CNI dynamically generated by tke-bridge-agent → race exists        |
| Overlay (VPC-CNI / GR)   | ❌ Not needed   | cilium fully owns CNI — kubelet won't create Pod sandboxes until cilium CNI is ready (natural) |

## How to configure

### Console

When creating a node pool, add the taint in **Advanced Settings**:

- Key: `node.cilium.io/agent-not-ready`
- Value: `true`
- Effect: `NoSchedule`

### Terraform

Regular node pool:

```hcl
resource "tencentcloud_kubernetes_node_pool" "cilium" {
  # ...
  taints {
    key    = "node.cilium.io/agent-not-ready"
    value  = "true"
    effect = "NoSchedule"
  }
}
```

Native node pool / Karpenter NodePool work similarly — set the taint via the corresponding resource's fields.
